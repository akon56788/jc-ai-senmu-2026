## Codex の標準ワークフロー

Codex が新規タスクに着手する際、以下のフローに従います。

### Step 0. PMO・補完実行モード判定

```
1. 通常セッションか、GitHub / Drive / Notion 更新を伴うセッションか確認
 - 通常：3行 PMO レポート
 - 更新あり：SSOT 差分・同期・担当ツールへの影響を詳細確認

2. Code / Cowork / Chat の補完が必要か判定
 - Token 不足、並行業務、一時離脱、確認待ちで主担当が止まっていないか
 - Codex が直接できる検証・差分確認・軽微な docs 修正か
 - 代替実行後に主担当へ戻せる粒度か

3. 補完実行する場合の原則
 - 主担当を置き換えず、一時的なサブ担当として動く
 - 実行済み内容、未決事項、戻し先を必ず記録する
 - 他ツールへの依頼文は「番号 + 日本語説明 + URL」形式を優先する
```

### Step 1. コンテキスト読み込み

```
1. SHARED_SOURCE.txt を読む
 → ツール間ルール、ファイル構成を確認

2. CLAUDE_enhanced.md.txt を読む
 → プロジェクト全体方針・ロードマップを把握

3. AGENTS.md.txt を読む
 → Codex の役割、LOM ルール、ファイル命名規則を確認

4. 作業内容に応じて以下を追加読込：
 □ CODEX_WORKFLOW.md.txt（このファイル）
 □ CODEX_PROJECT_SETUP.txt（環境セットアップが必要な場合）
 □ VISION.md.txt（詳細戦略参照が必要な場合）
```

### Step 2. 作業計画立案

```
1. ユーザーからの要件を整理
 - 修正対象：Word / Excel / Markdown？
 - 修正内容：VBA / フィールド / 形式？
 - 検証方法：何で品質確認する？
 - 主担当：Code / Cowork / Chat / Codex のどれか？
 - Codex は主担当か、補完・代替・保管か？

2. 成果物の命名規則を決定
 例：2026-05-05_lom_template_draft.docx

3. 環境確認
 - ローカルプロジェクトのディレクトリ構成確認
 - templates/ から基盤ファイルを確認
 - Drive から最新コンテキストを確認
```

### Step 2-補足. Code / Cowork 補完時の引き継ぎ

Codex が Code または Cowork の代わりに確認・実行した場合、完了報告には以下を含めます。

```markdown
【補完対象】Code / Cowork / Chat
【Codex実施済み】確認・編集・検証した内容
【結果】OK / 差分あり / 要判断
【戻し先】次に主担当がやること
【参照】PR、Drive URL、Notion項目、対象ファイル
```

補完実行の例：

- Code の代替：PR差分確認、Actions結果確認、docs の軽微修正、GitHub → Drive 同期確認
- Cowork の代替：Notion登録前のタスク整理、Driveリンク確認、進捗報告文の下書き
- Chat の補助：論点整理、リスク整理、次スレッド命名案の作成

### Step 3. ローカル環境での修正

```
1. templates/ から基盤ファイルをコピー
 例：lom_base_template.xlsx → outputs/ にコピー

2. outputs/ 内で修正作業開始（_draft 版）
 例：2026-05-05_lom_template_draft.xlsx

3. 修正内容：
 ✓ VBA マクロの追加・修正
 ✓ Word フィールドコード実装
 ✓ セル保護・シート保護設定
 ✓ フォーマット統一
```

### Step 4. 検証・テスト

```
1. ファイル種別に応じた検証チェックリスト実行
 → 下記「ファイル種別別検証チェックリスト」参照

2. テストデータでの動作確認
 - Excel：サンプルデータ入力 → マクロ動作確認
 - Word：フィールド自動入力テスト
 - 結果：scratch/ に test_result_*.md 記録

3. 問題があれば修正・再検証
```

### Step 5. 最終準備

```
1. ファイル名を _final に変更
 例：2026-05-05_lom_template_draft.xlsx
 → 2026-05-05_lom_template_final.xlsx

2. 検証レポート作成
 例：2026-05-05_lom_template_validation_report.md
 内容：
 - テスト日時
 - テスト項目と結果（✓ / ✗）
 - 修正内容
 - 品質判定（OK / 要改修）

3. Drive / GitHub へアップロード準備
 - _final ファイル
 - 検証レポート
```

### Step 6. Drive / GitHub 反映・報告

```
1. Codex または Code が outputs/ 内の _final ファイルをアップロード
 - Phase 0：Google Drive へアップロード
 - Phase 1：GitHub リポジトリにコミット

2. ユーザーに報告
 - 成果物ファイル名
 - 検証結果
 - 次ステップ（Cowork 実行待ち等）

3. ローカルで記録（scratch/ に作業メモを保存、後で削除可）
```

## ファイル命名規則（詳細）

### 基本形式

```
YYYY-MM-DD_[プロジェクト]_[内容]_[ステータス].[拡張子]
```

### 各要素の説明

| 要素 | 説明 | 例 |
|------|------|---|
| `YYYY-MM-DD` | 作成日（ISO 8601） | `2026-05-05` |
| `[プロジェクト]` | 対象プロジェクト | `lom`, `template`, `checklist` |
| `[内容]` | 修正・生成内容 | `template`, `validation`, `report` |
| `[ステータス]` | ファイルの段階 | `draft`, `final`, _(オプション)_ |
| `[拡張子]` | ファイル形式 | `.xlsx`, `.docx`, `.md` |

### 具体例

```
LOM テンプレート生成：
 - 2026-05-05_lom_template_draft.xlsx
 - 2026-05-05_lom_template_final.xlsx

検証レポート：
 - 2026-05-05_lom_template_validation_report.md

チェックリスト：
 - 2026-05-05_lom_checklist_draft.docx
 - 2026-05-05_lom_checklist_final.docx

その他テンプレート：
 - 2026-05-05_word_form_template_draft.docx
 - 2026-05-05_markdown_template_final.md

GitHub Actions ワークフロー：
 - 2026-05-05_github_actions_sync_workflow_draft.yml
 - 2026-05-05_github_actions_sync_workflow_final.yml
```

### ファイル配置ルール

| 段階 | 配置場所 | 例 |
|------|---------|---|
| **作業中** | outputs/ + _draft 名 | outputs/2026-05-05_lom_template_**draft**.xlsx |
| **完成** | outputs/ + _final 名 | outputs/2026-05-05_lom_template_**final**.xlsx |
| **Drive 送信** | Google Drive | —（_final ファイルのみ） |
| **GitHub コミット** | GitHub / docs フォルダ | github.com/.../docs/CODEX_WORKFLOW.md.txt |
| **一時メモ** | scratch/ | scratch/notes.md |

## ファイル種別別検証チェックリスト

### Excel ファイル（.xlsx）検証チェックリスト

```
【基本検証】
□ ファイルを Excel で開ける
□ シート数が正しい（計 N シート）
□ セル保護が設定されている
□ パスワード保護なし（ユーザーが簡単に編集可能）

【機能検証】
□ VBA マクロが有効化できる
□ 日付自動挿入マクロ → テスト
□ チェック機能（ある場合） → テスト
□ 計算式が正しく動作する（SUM、IF 等）

【データ検証】
□ サンプルデータを入力
□ 計算結果が期待値と一致
□ フォーマット（セル色、フォント）が統一
□ 小数点・通貨表示が正しい

【LOM 専用検証】
□ Round 番号が正しい（1〜7、4 スキップ）
□ 月度が自動入力される
□ 見出し・セル配置が LOM 標準に準拠

【ファイル確認】
□ ファイルサイズが妥当（5MB 以下推奨）
□ 最終セーブ時刻が本日または昨日
□ ファイル名が命名規則に従っている
```

### Word ファイル（.docx）検証チェックリスト

```
【基本検証】
□ ファイルを Word で開ける
□ 見出し数が正しい
□ ページ数が予定範囲（LOM = 2〜4 ページ）

【フィールドコード検証】
□ 日付フィールド → 自動更新される
□ 回数フィールド（LOM Round） → 自動更新される
□ 月度フィールド → 自動更新される

【フォーマット検証】
□ 見出し：明朝、12pt、太字
□ 本文：ゴシック、11pt、行間 1.5
□ インデント：1行目 0cm、左右 0cm
□ ページ余白：上下左右 20mm

【スペル・文法検証】
□ スペルチェック：エラーなし
□ 文法チェック（言語が日本語） → エラーなし
□ 用語統一：同じ概念は同じ用語で

【LOM 専用検証】
□ 見出し構成が LOM 標準に準拠
□ 敬語・言葉遣いが適切
□ 日付が月初（初日）に設定可能

【ページブレーク検証】
□ ページブレークが正しい位置にある
□ 孤立した見出し（ページ最下部）がない
□ 表が途中で分割されていない

【ファイル確認】
□ ファイルサイズが妥当（3MB 以下推奨）
□ 最終セーブ時刻が本日または昨日
□ ファイル名が命名規則に従っている
□ 修正履歴が削除されている（必要に応じて）
```

### Markdown ファイル（.md）検証チェックリスト

```
【基本検証】
□ テキストエディタで開ける
□ 文字エンコーディング：UTF-8

【Markdown 文法検証】
□ ヘッダ記号 (#) が正しい（# H1, ## H2, ### H3）
□ リスト（- または 1.）が正しくインデント
□ コードブロック（```）が閉じられている
□ リンク記法：[テキスト](URL) 形式

【内容検証】
□ 見出しが階層的（H1 → H2 → H3）
□ 文章が簡潔（1 段落 3〜5 文）
□ 用語が統一
□ 参照リンク（URL）が有効

【テンプレート専用検証】
□ プレースホルダ {{}} 記法が統一
□ コメント \<\!-- --\> 形式で説明あり
□ セクション区切り（---）が適切

【ファイル確認】
□ ファイルサイズが妥当（1MB 以下推奨）
□ 最終セーブ時刻が本日または昨日
□ ファイル名が命名規則に従っている
```

### GitHub Actions YAML（.yml）検証チェックリスト

```
【YAML 文法検証】
□ インデント：2 スペース統一
□ コロン（:）後は 1 スペース
□ ブロック記号（|-）が正しい
□ 引用符（'/"）が対応している

【Workflow 構造検証】
□ name：ワークフロー名が明確
□ on：トリガー定義（schedule / push / workflow_dispatch）
□ jobs：ジョブ構成（ubuntu-latest 等）
□ steps：各ステップの順序が論理的

【GHub Actions アクション検証】
□ actions/checkout@v3 で最新版を使用
□ アクション名・バージョンが GitHub Marketplace に存在
□ with: パラメータが正しい

【認証・シークレット検証】
□ GitHub Secrets 参照形式：\${{ secrets.SECRET_NAME }}
□ Service Account JSON：base64 エンコード確認
□ Token 有効期限（期限切れ X）

【Drive API / GitHub MCP 検証】
□ Google Drive API 認証方式（OAuth / Service Account）
□ GitHub MCP エンドポイント有効
□ Dry-run 実行：エラーなし
□ ログ出力：敏感情報がマスクされている

【ファイル確認】
□ ファイルサイズが妥当（20KB 以下推奨）
□ ファイル名が規則に従っている
□ リポジトリの正しいパス（.github/workflows/）
```

## トラブルシューティング

### Excel VBA が動作しない

```
症状：マクロ実行時に「マクロが見つかりません」エラー

対応：
1. Excel で Visual Basic エディタ（Alt + F11）を開く
2. VBA コード確認 → 構文エラーがないか確認
3. マクロ セキュリティ設定：
 - 信頼できる場所に ファイルを移動
 - または「開発」タブ → マクロ セキュリティ を「すべてのマクロを有効」に
```

### Word フィールドが自動更新されない

```
症状：日付フィールドが { DATE } のまま表示

対応：
1. Word でそのフィールドを選択
2. Ctrl + Shift + F9 で全フィールド更新
3. またはフィールド右クリック → 更新
```

### ファイルが Drive / GitHub にアップロードできない

```
症状：Drive MCP / GitHub MCP エラー、または認証失敗

対応：
1. ファイルサイズ確認（5MB 以下なら OK）
2. 認証情報確認：
 - Service Account JSON の有効期限
 - GitHub Token の有効期限
 - ローカルの認証キャッシュ削除
3. 手動アップロード推奨：
 - Drive ブラウザで「ファイルをアップロード」
 - または GitHub Web UI で直接アップロード
```

### GitHub Actions ワークフロー実行エラー

```
症状：GitHub Actions ジョブ失敗、Drive API 認証エラー

対応：
1. GitHub Actions ログ確認
2. Service Account JSON が Base64 エンコードされているか確認
3. GitHub Secrets に正しい値が登録されているか確認
4. Dry-run で動作テスト（manual dispatch）
5. API レート制限確認（Google Drive / GitHub）
```
