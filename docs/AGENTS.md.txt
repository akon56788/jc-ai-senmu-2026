## Codex の補完・代替ロール（Phase 4.5+）

Codex は Agent PMO に加えて、Code と Cowork の **サブ担当（並行・代替・保管）** としても機能します。これは主担当を置き換えるものではなく、Token 不足・並行業務・確認待ち・一時停止が発生したときに、プロジェクトを止めないための補助役です。

### 補完対象

| 対象 | Codex が担う補完内容 | 主担当への戻し方 |
|------|---------------------|----------------|
| **Code** | GitHub 差分確認、PR / Actions 確認、軽微な docs 修正、検証結果の整理、実装案レビュー | 変更内容・検証結果・未決事項を Code 向け依頼文で返す |
| **Cowork** | Notion / Drive / Sheets 連携状況の確認、進捗報告文の作成、未登録・遅延タスクの洗い出し | Cowork が実行しやすい形式で対象・期限・根拠リンクを渡す |
| **Chat** | 方針相談に必要な論点整理、スレッド命名案、リスク・選択肢の下書き | Chat が判断できるように前提・選択肢・推奨を短く渡す |

### 実行条件

- 主担当ツールの Token / 時間 / 並行作業制約で処理が滞る場合
- GitHub / Drive / Notion の同期確認など、Codex が直接検証できる作業がある場合
- ユーザーが「Codexで代わりに」「並行で」「保管して」「引き継ぎ用に」と依頼した場合
- PMO チェックでボトルネックを検出し、短時間の補完で解消できる場合

### 制約

- 主担当の責務・判断権限を恒常的に奪わない
- Notion や Drive の本番更新は、権限と同期ルールを確認してから実行する
- 代替実行した場合は、必ず「何を実行したか」「どこまで終わったか」「誰に戻すか」を残す
- 競合しそうなファイルは、編集前に Git / Drive の最新状態を確認する

### 出力フォーマット

人を介して他ツールに依頼・引き継ぎする場合は、以下の形式を使います。

```markdown
【依頼先】Code / Cowork / Chat
【目的】何を完了・確認してほしいか
【背景】Codex が確認した事実・差分・リスク
【対象】ファイル名、PR、Drive URL、Notion項目など
【依頼内容】番号付きで具体化
【Codex実施済み】実行済み作業・検証結果
【未決事項】判断待ち・確認待ち
```

## ChatGPT / Gemini CLI 外部レビューの扱い（Phase 4.5+）

重要な方針変更、SSOT更新、GitHub / Drive / Notion 連携、ツール責務変更では、Codex PMO は外部レビューを依頼できます。

- ChatGPT は手動・対話型の外部顧問レビュー役
- Gemini CLI は Codex から半自動で呼び出すレビュー実行役
- どちらの結果も助言であり、最終判断・GitHub反映・Drive SSOT反映は行わない
- Codex はレビュー結果を `Blocker` / `Risk` / `Suggestion` / `No issue` に分類し、ユーザー確認用サマリーを作成する
- ユーザー承認前に Drive SSOT、Notion、本番運用へ反映しない
- 詳細な ChatGPT 役割定義は `CONTEXT_FOR_CHATGPT.md` を参照する

ChatGPT にレビュー依頼する場合、Codex は必要な SSOT 要約・差分・判断対象を依頼文に含めます。

## スレッド命名規則（全ツール共通・標準化）

プロジェクトの進捗を「スレッド」として記録する際、以下の命名規則を統一します。

### 基本形式

```
[ステータス絵文字] 【[フェーズ/プロジェクト名]】[メインタイトル]
[詳細説明（成果物・付属作業）]
（[使用モデル]）
```

### 要素解説

| 要素 | 説明 | 例 |
|------|------|---|
| **ステータス絵文字** | ✅ = 完成・完了 / 🔄 = 進行中 / 📋 = 計画・提案 / ⚠️ = 課題・調整中 | ✅ |
| **フェーズ/プロジェクト名** | Phase 番号または機能エリア名 | Phase 0、Phase 1、Notion PoC |
| **メインタイトル** | 何が完成したか/進行中か（簡潔に） | コンテキストファイル展開完成 |
| **詳細説明** | 成果物・付属作業を箇条書き的に列挙 | 5 つの .txt ファイル作成・outputs 配置 |
| **使用モデル** | Claude の場合はモデル名、その他は対象ツール | Claude Haiku 4.5 / Codex / Cowork |

### 具体例

#### 例 1（完成系）
```
✅ 【Phase 0 コンテキストファイル展開完成】
Google Drive .gdoc 自動変換対策 + 5 つの .txt ファイル作成・outputs 配置・old フォルダ構成設計完了
（Claude Haiku 4.5）
```

#### 例 2（進行中）
```
🔄 【Phase 1 GitHub 統合準備中】
リポジトリ初期化 + GitHub Actions テンプレート作成・MCP 設定検証中
（Claude Code / Codex）
```

#### 例 3（計画段階）
```
📋 【Notion PoC 拡張構想】
新規 Committee Database / タスク管理 DB 統合 / Slack 連携検討
（Claude Chat / Cowork）
```

#### 例 4（課題対応）
```
⚠️ 【LOM テンプレート VBA デバッグ中】
Round 5 日付自動入力マクロ不具合・回避策検証中
（Codex）
```

#### 例 5（Phase 1 GitHub 関連）
```
🔄 【Phase 1 GitHub Actions テンプレート作成中】
GitHub MCP 設定 + 毎日テスト実行パイプライン・Drive 自動同期スクリプト設計中
（Claude Code / Codex）
```

### ツール別 推奨タイトルパターン

#### Claude Chat → Code への仕事指示
```

仕様設計 + Code 向け実装ガイド作成
```

**特徴**：输出形式（Word / メール等）を明記

## LOM 発送ルール（Cowork・Codex 向け重要）

### LOM とは

\"Letter of Management\" — 専務理事から各委員会への月次通達。毎月 1 回発送。

### 発送スケジュール

| 回 | 月 | 発送目安 | 備考 |
|----|---|---------|-----|
| 1 | 1 月 | 上旬 | 新年方針 |
| 2 | 2 月 | 上旬 | — |
| 3 | 3 月 | 上旬 | — |
| \~\~4\~\~ | \~\~4 月\~\~ | — | **スキップ** (新年度、別途対応) |
| 5 | 5 月 | 上旬 | — |
| 6 | 6 月 | 上旬 | — |
| 7 | 7 月 | 上旬 | — |

**合計 7 回**（4 月 = スキップ）

### テンプレート・チェックリスト

#### Excel テンプレート仕様

| 項目 | 要件 |
|------|------|
| ファイル名 | `lom_template_round_[N]_[YYYY-MM].xlsx` |
| シート数 | 2（メイン、チェックリスト） |
| セル形式 | 保護あり、入力セルのみ編集可 |
| VBA | チェック機能、日付自動挿入 |

#### Word テンプレート仕様

| 項目 | 要件 |
|------|------|
| ファイル名 | `lom_round_[N]_[YYYY-MM].docx` |
| フォーマット | 見出し：明朝、本文：ゴシック |
| フィールドコード | 日付・回数・月度の自動挿入 |
| ページ数 | 通常 2〜4 ページ |

#### 発送前チェックリスト（Codex / Cowork 向け）

```
□ Excel：
 ✓ セル保護が正しく設定されているか
 ✓ VBA マクロが動作するか（日付自動入力等）
 ✓ サンプルデータで動作確認
 ✓ ファイル名が規則に従っているか

□ Word：
 ✓ フィールドコード（日付・回数）が自動入力されるか
 ✓ スペルチェック
 ✓ フォーマット統一（フォント・行間）
 ✓ ページブレークが正しいか

□ 納品：
 ✓ ファイル名が規則に従っているか
 ✓ Drive の正しいフォルダにアップロード
 ✓ Sheets 進捗ダッシュボード更新
```

## ファイル命名規則

### コンテキストファイル（Google Drive）

```
[DOMAIN].txt

例：
 - SHARED_SOURCE.txt
 - CLAUDE_enhanced.md.txt
 - AGENTS.md.txt
```

### 成果物ファイル（outputs / Drive）

```
YYYY-MM-DD_[プロジェクト]_[内容]_[ステータス].[拡張子]

例：
 - 2026-05-05_lom_template_draft.docx
 - 2026-05-05_lom_template_final.xlsx
 - 2026-05-05_lom_validation_report.md
```

**ステータス**：

- `draft` — 作業中、未検証（outputs のみ、Drive 非同期）
- `final` — 完成、検証済み（Drive にアップロード）
- `report` — 検証レポート（Drive にアップロード、参考用）

## よくある質問

### Q. Cowork が LOM テンプレートを改修したい場合は？

**A.** Codex に依頼してください。Cowork は既存テンプレートの展開・操作のみ。改修は Codex が担当。

### Q. Code が LOM Excel を生成できる？

**A.** 初版テンプレートなら可能。ただし VBA マクロ実装は Codex に。

### Q. Chat が LOM 方針を変える場合は？

**A.** AGENTS.md.txt の「LOM 発送ルール」セクションを更新 → Code が実施。

### Q. ChatGPT で LOM 文案を作成できる？

**A.** はい。ただし ChatGPT が生成した文案は Chat / Code で品質チェック後、Word に統合。

### Q. Phase 1 で Drive から GitHub に移行する？

**A.** 段階的移行です。Phase 1 では GitHub を更新起点・版管理の中心にし、確定した内容をDriveへ反映します。Drive は引き続き各ツールの参照元として維持します。詳細は DRIVE_SYNC_STATUS.md（リポジトリ内）を参照。
